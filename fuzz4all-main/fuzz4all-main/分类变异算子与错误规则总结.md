# 分类变异算子与错误规则总结
profile_loader.py

## 1. 系统里“规则”分几层

当前规则大致分成三层：

1. `mutation_profile / priority_operators`
   - 决定某个 API 分类后，优先使用哪些变异算子。
2. `repair_rules / filter_rules`
   - 决定 seed 或 mutation 在进入编译前，自动补什么 import、过滤哪些明显垃圾模式。
3. `extract_javac_error_categories() + map_error_categories_to_rules()`
   - 决定编译/运行失败后，如何把错误归类，并转成下一轮 prompt 的约束规则。

所以你可以把系统理解成：

- **分类层**：决定这是什么类 API
- **变异层**：决定用哪些 operator 改代码
- **修复层**：决定哪些 import/结构要补、哪些垃圾模式要拦
- **反馈层**：决定失败后下一轮该提醒模型什么

## 3. 当前通用错误类别（反馈层）

当前 `JAVA.py` 已识别的通用错误类别包括：

- `nonexistent_method`
- `undeclared_variable`
- `protected_member_access`
- `write_used_as_expression`
- `invalid_write_overload`
- `invalid_overload_call`
- `invalid_method_signature`
- `invalid_constructor`
- `unchecked_ioexception`
- `unchecked_exception`
- `lossy_conversion`
- `public_class_filename_mismatch`
- `ambiguous_reference`
- `incomplete_structure`
- `invalid_try_structure`
- `invalid_statement`
- `invalid_exception_handling`
- `invented_buffer_constant`
- `internal_jdk_api`
- `weaker_override_access`
- `invalid_constructor_argument`
- `reflection_api_misuse`
- `missing_import`
- `incompatible_types`
- `invalid_generic_syntax`
- `nonexistent_type`
- `non_static_context`
- `void_assignment`

这些错误会被映射成下一轮 prompt 里的通用规则，例如：

- 变量必须先声明再使用
- 只用文档中存在的方法和类型
- 参数个数和类型必须匹配
- checked exception 要放进 `try/catch`
- 反射要用正确 receiver 和参数类型数组
- 不要把 void 返回值赋给变量

## 4.5 CONCURRENT

### 主要变异算子

- `concurrent_yield_injection`
- `concurrent_thread_wrapper_mutation`
- `concurrent_signal_sequence_mutation`

### repair 规则

- `byte_literal_cast = True`
- 正面构造器提示 `api_positive_hints`：
  - 优先使用最简单、文档化的构造器
  - functional-interface 参数先绑定到具名变量，再传给构造器

### filter 规则

#### reject_tokens

- `Thread.Priority`
- `setDefaultPriority`

#### reject_patterns

- `while(true)`
- `for(;;)`
- `Thread.sleep(Long.MAX_VALUE`
- `new Thread(..., "name", int)` 这类三参误构造
- `new Thread(...).setPriority(...)` 这类链式误用

### 分类级反馈规则

- 线程/interleaving 结构要保持语法隔离
- 优先级只能通过 `setPriority(...)` + `Thread` 常量设置
- 构造、配置、启动、`sleep/join/wait` 要拆成独立语句
- `non_static_context` 时：
  - 线程实例必须在 `main()` 中本地创建
- `void_assignment` 时：
  - 不要把 `setPriority/setDaemon` 等链式写进构造表达式

## 5. 当前更值得关注的“seed / mutation 错误规则”重点

从当前代码和近期实验来看，后续最值得继续收敛的重点是：

### `CONCURRENT`

- 非法构造器组合
- 把 `join()` 当成有返回值
- 把 `Runnable` 错当成 `ThreadGroup`
- 需要更多正面合法构造器示例

### `REFLECT`

- `Type[]` 和 `Class<?>[]` 混淆
- 反射签名参数数组结构错误
- `invoke(null, ...)` 导致 receiver 空指针

### `SECURITY`

- `Cipher.init(...)` 路径
- `wrap/unwrap/doFinal` 的合法对象关系
- checked exception / key / spec / provider 语义约束

### `CALLBACK`

- listener lambda 参数签名不匹配
- add/remove/fire 的对象归属和调用顺序

### `JVM_MGMT`

- prompt 入口质量仍然非常关键
- getter/query 路径必须沿着标准 MXBean 接口走，不能再猜方法名
