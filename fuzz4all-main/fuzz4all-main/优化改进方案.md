## 总目标

**把系统从“能按分类跑起来”推进到“按分类稳定地产生有效样本”。**

也就是说，现在不是再重做架构，而是做最后的**分类级收敛**：

- 保留现有的文档输入、离线分类、分类驱动变异、反馈刷新闭环
- 通过不断运行发现高频失败模式
- 把这些失败模式沉淀成**通用规则**和**分类规则**
- 让每个类别的 seed 和 mutation 更稳定，最终提高 SAFE 率

---

## 这套方案的核心思想

### 1. 不改主架构
因为现在主架构已经成立了：

- API 文档输入
- 离线五维分类
- target 配置
- 分类驱动 `mutation_profile`
- 大模型生成 seed
- 分类算子做 mutation
- 编译/运行验证
- 错误提炼成规则
- prompt refresh 闭环

这些不是当前瓶颈，所以方案目的**不是重做这一套**。

---

### 2. 主要做“分类级收敛”
现在的问题是：

- 分类已经命中了
- 变异策略已经按分类生效了
- 但某些类别效果还不稳

所以方案核心不是“再加更多 generic 规则”，而是：

**把每个类别里重复出现的错误模式，收敛成该类别自己的规则。**

例如：
- 并发类总是错用构造器和优先级
- JVM 管理类总是猜错 getter 名或 proxy 路径
- 反射类总是签名和辅助类型不稳

方案目的就是：
- 找出这些高频结构性错误
- 用分类级规则把它们压下去

---

### 3. 尽量避免单 API 特判
这点非常关键。

我的方案一直坚持的是：

- 优先加**通用规则**
- 再加**分类规则**
- 最后才允许很少量 target 级增强做验证

而不是：
- 因为某个 API 报错了，就写一个只对这个 API 生效的 if/else

所以整体目标不是“把几个代表 API 调通”，而是：

**让同一类 API 都能受益。**

## 这套方案当前在做什么

### 第一步：通过运行找高频失败模式
例如：
- `Thread` 总卡非法构造器
- `ManagementFactory` 总猜错 getter
- `Method` 总在反射签名和辅助类型上出错

### 第二步：把高频错误分两层沉淀
#### 通用规则
例如：
- 变量先声明
- 不要把 void 赋值
- 参数类型要匹配
- checked exception 要处理

#### 分类规则
例如：
- 并发类：优先级只能通过实例方法设置，构造/配置/启动分开
- JVM 管理类：只用标准 MXBean 接口 getter，不猜 `*Information/*Count`
- 反射类：参数签名必须是标准 `Class` / `Class[]`

### 第三步：把这些规则回写到系统中
回写的位置通常有三类：

1. `repair/import` 规则  
2. `filter/reject` 规则  
3. `feedback -> 下一轮 prompt` 的约束规则  

### 第四步：再运行验证是否起效
不是只看一轮，而是看：
- SAFE 有没有起来
- 失败模式有没有改变
- 某类问题是不是从“完全乱飞”变成“标准路径上的细节错误”

## 这套方案最终想达到什么效果

### 机制层
已经做到并希望保持：
- 10 类代表 API 都命中正确分类
- 都能加载对应 `mutation_profile`
- 都能调用对应分类的变异算子
- 都能进入 feedback/refresh 闭环

### 效果层
想进一步做到：
- 强类稳定
- 弱类不再是 0 SAFE
- 波动类逐渐稳定
- 不同类别都能在自己的语义路径上产生更多可编译/可运行样本

也就是说，最终目标不是：
- “分类做对了”

而是：

**分类不仅做对了，而且真的把代码生成和变异质量拉起来了。**

## 对目前几个类别分别的目标

### `CONCURRENT`
目的：
- 压掉非法构造器和优先级误用
- 让并发类从结构性失败转向合法线程生命周期代码

### `JVM_MGMT`
目的：
- 不再猜管理类 getter
- 让代码集中到标准 MXBean/query 路径上
- 从 0 SAFE 拉到持续有 SAFE

### `REFLECT`
目的：
- 稳定反射签名、receiver、辅助类型
- 减少模板级幻觉和导入问题
- 让反射类从“偶尔成功”变成“稳定有成功样本”
