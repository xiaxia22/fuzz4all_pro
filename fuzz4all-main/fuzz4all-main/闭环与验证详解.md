# Java API 定向 Fuzz 系统闭环与验证详解

## 文档用途

这份文档的目标不是简单总结“做了什么”，而是把整套系统从输入、分类、生成、变异、编译/运行验证、反馈提炼、prompt 更新到再次生成的**完整闭环**讲清楚。

你可以把它当成：

- 组会答疑稿
- PPT 讲稿补充说明
- 老师追问“具体怎么实现”的备忘
- 后续写中期/结题材料时的技术说明底稿

---

# 一、系统目标到底是什么

这套系统要解决的问题不是“让大模型随便生成 Java 代码”，而是：

1. 先让系统知道某个 Java API 属于什么类别
2. 再让系统按这个类别去选择合适的 fuzz 策略
3. 生成第一批 seed 代码
4. 对 seed 做分类感知的变异
5. 将 seed 和 mutation 交给编译器/运行时验证
6. 把错误信息总结成下一轮提示词约束
7. 再继续生成、变异和验证

所以这不是单次生成任务，而是一个**可迭代优化的闭环系统**。

---

# 二、完整闭环总览

可以把系统理解成下面这条主链：

```text
API 文档
  ↓
离线五维分类
  ↓
target 配置
  ↓
初始 prompt 生成
  ↓
LLM 生成 seed
  ↓
分类驱动 mutation
  ↓
javac / java 验证
  ↓
错误归因与规则提炼
  ↓
runtime prompt 更新 / refresh
  ↓
进入下一轮生成
```

# 三、每一步到底做了什么

## 1. API 文档输入层

### 使用的目录

- `config/documentation`

### 这里存的是什么

这里不是“Java 新特性说明”，而是**每个 API 一份独立文档**。

例如：

- `java.base_javax_crypto_Cipher.md`
- `java.base_java_io_File.md`
- `java.base_java_lang_reflect_Method.md`

### 为什么要这样设计

因为系统后面所有关键步骤都依赖 API 语义：

- 离线分类要读它
- 初始 prompt 要读它
- target 配置要指向它

如果输入仍然是泛语言特性文档，那么后续系统就无法真正做到“按 API 定向 fuzz”。

### 如何验证这一步已经完成

检查：

- `config/targeted/*.yaml` 里的 `path_documentation` 是否都指向 `config/documentation/*.md`

如果 target 配置已经绑定 API 文档，说明输入层已经切换成功。

---

## 2. 离线五维分类层

### 使用的代码和结果目录

- 分类脚本：`scripts/classify_java_apis.py`
- 分类结果：
  - `outputs/api_classification/java_api_5d_classification.json`
  - `outputs/api_classification/java_api_5d_classification.csv`
  - `outputs/api_classification/java_api_5d_summary.md`

### 这一步做了什么

对每个 API 文档做一次离线分析，并输出：

- `primary_tag`：主分类标签
- `all_tags`：辅助标签集合
- `mutation_profile`：建议的变异画像
- `priority_operators`：优先使用的变异算子
- `risk_points`：该类 API 重点关注的边界/风险点

### 当前主分类标签

系统当前支持的主分类包括：

- `SECURITY`
- `FILE`
- `REFLECT`
- `CONCURRENT`
- `CALLBACK`
- `TIME`
- `NETWORK`
- `UTILITY`
- `JVM_MGMT`
- `RESOURCE`

辅助结构标签包括：

- `BUFFER`
- `BATCH_OP`
- `MARK_SUPPORT`
- `GENERIC`

### 为什么要先离线分类

因为你这个系统的目标不是对所有 API 用同一种变异方法，而是：

- 安全类更关注算法、provider、密钥、参数
- 文件类更关注路径、状态、权限
- 反射类更关注成员解析、receiver、accessibility
- 并发类更关注生命周期、顺序、调度

离线分类就是把这些先验知识固定下来。

### 如何验证这一步已经完成

检查 `java_api_5d_classification.json`，看目标 API 是否已经带有：

- `primary_tag`
- `mutation_profile`

例如：

- `Cipher -> SECURITY`
- `File -> FILE`
- `Method -> REFLECT`
- `Thread -> CONCURRENT`

如果这些能稳定命中，说明离线分类层已经工作。

## 3. target 配置层

### 使用的目录

- `config/targeted`

### 这一步做了什么

为每个 API 生成一份单独的 yaml 配置，里面包括：

- `path_documentation`
- `target_string`
- `enable_mutation`
- `path_classification_result`
- `mutation_operator_topk`
- `prompt_refresh_interval`
- `autoprompt` 相关参数

### 为什么需要这一层

因为系统后续是按“单个 API 任务”运行的，每个 API 都需要一套独立配置来绑定：

- 文档
- 分类结果
- 输出目录
- 运行参数

### 如何验证这一步已经完成

查看某个配置，例如：

- `config/targeted/java.base_javax_crypto_Cipher.yaml`

检查是否具备：

- `path_documentation`
- `path_classification_result`
- `enable_mutation: true`

如果具备，说明 target 层已经能把文档和分类结果带入运行时。

## 4. 分类到运行时 profile 的映射层

### 使用的代码

- `Fuzz4All/mutation/profile_loader.py`

### 这一步做了什么

读取离线分类结果，为当前 API 组装一份运行时 profile，主要包括：

- `primary_tag`
- `mutation_profile`
- `active_profiles`
- `priority_operators`
- `repair_rules`
- `filter_rules`

### 举例说明

例如：

- `Cipher`
  - `primary_tag = SECURITY`
  - `mutation_profile = security_provider_flow`
- `File`
  - `primary_tag = FILE`
  - `mutation_profile = file_path_state`
- `ManagementFactory`
  - `primary_tag = JVM_MGMT`
  - `mutation_profile = jvm_mgmt_runtime_state`

### 这一步为什么重要

因为离线分类结果如果不能进入运行时，那就只是“贴标签”，并不能真正影响生成与变异。

`profile_loader.py` 是把分类结果转成运行时行为的桥梁。

### repair_rules 和 filter_rules 是什么

#### `repair_rules`

用于自动补一些低成本修复，例如：

- 自动补 import
- byte literal cast

#### `filter_rules`

用于提前过滤高概率错误或幻觉模式，例如：

- 伪常量
- 非法 helper 名称
- 明显非法语法片段

### 如何验证这一步已经完成

看运行后的 `log.txt`，系统会打印类似：

- `Mutation enabled with primary tag ...`
- `profile ...`
- `operators [...]`

如果这里和预期分类一致，说明 profile loader 已经成功驱动运行时。

---

## 5. 初始 prompt 生成层

### 使用的代码

- `Fuzz4All/target/target.py`

### 这一步做了什么

系统会根据 API 文档生成初始 prompt。

当前方式是：

1. 从文档读取 API 语义
2. 生成多个候选 prompt
3. 对候选 prompt 进行简单评分
4. 选出 `best_prompt.txt`

### 输出文件

- `prompts/best_prompt.txt`
- `prompts/greedy_prompt.txt`
- `prompts/prompt_0.txt`
- `prompts/prompt_1.txt`
- `prompts/prompt_2.txt`
- `prompts/scores.txt`

### 为什么需要这一步

因为不同 API 的初始 seed 提示词不能完全一样。  
例如安全类 prompt 和反射类 prompt 的侧重点本来就不同。

### 如何验证这一步已经完成

看输出目录的 `prompts/` 文件夹里是否已经出现：

- `best_prompt.txt`
- `scores.txt`

如果出现，说明初始 prompt 生成阶段已经跑完。

---

## 6. seed 生成层

### 使用的模型

- 代码生成模型：Qwen / Ollama（本地）

### 这一步做了什么

根据 `best_prompt.txt`，生成第一批 Java seed 代码。

seed 的角色是：

- 作为第一批候选样本
- 也是后续 mutation 的起点

### 为什么需要 seed

因为 mutation 不是凭空开始的，而是先有一批初始代码，再围绕其做边界扰动。

### 如何验证这一步已经完成

查看输出目录是否产生：

- `0.fuzz`
- `1.fuzz`
- 其他样本文件

如果生成了 `.fuzz` 文件，就说明 seed 已经产生。

---

## 7. 分类驱动 mutation 层

### 使用的代码

- `Fuzz4All/mutation/java_mutator.py`

### 这一步做了什么

系统根据当前 API 的 `mutation_profile` 选择 operator，对 seed 做定向变异。

### operator 是什么

operator 就是具体的变异算子，例如：

- `security_provider_mutation`
- `file_path_traversal_mutation`
- `reflect_accessibility_flip_mutation`
- `callback_duplicate_registration_mutation`
- `jvm_mgmt_query_name_mutation`

### 为什么这一步是系统核心

因为这一步决定了系统是不是“分类感知”的：

- 如果所有 API 都只用 generic mutation，那设计目标没实现
- 只有当不同类别真的用了不同 operator，分类才真正起作用

### 如何验证这一步已经完成

查看：

- `mutation_manifest.jsonl`

里面每条记录会有：

- `operator`
- `source_type`
- `primary_tag`
- `mutation_profile`

如果你能看到：

- `security_*`
- `file_*`
- `reflect_*`
- `callback_*`
- `jvm_mgmt_*`

说明分类驱动变异已经生效。

---

## 8. 编译与运行验证层

### 使用的代码

- `Fuzz4All/target/JAVA/JAVA.py`

### 这一步做了什么

把 seed 和 mutation 后的代码分别交给：

- `javac`
- 必要时交给 `java`

然后根据结果把样本分成：

- `SAFE`
- `FAILURE`
- `ERROR`

### 这三类分别是什么意思

#### `SAFE`

- 编译成功
- 运行也结束

表示这是当前最直接的“有效样本”。

#### `FAILURE`

- 编译失败

通常意味着：

- 语法错误
- import 缺失
- API 调用错误
- 参数个数或类型不匹配

#### `ERROR`

- 编译通过，但运行时报错

这不一定是坏事，因为说明：

- 样本已经进入可执行路径
- 比单纯编译失败更接近“深层行为”

### 如何验证这一步已经完成

查看：

- `log_validation.txt`
- `mutation_manifest.jsonl`

并统计其中的：

- `SAFE`
- `FAILURE`
- `ERROR`

---

## 9. 错误归因与规则提炼层

### 使用的代码

- `Fuzz4All/target/JAVA/JAVA.py`

### 这一步做了什么

系统不是把原始 javac 错误直接丢给模型，而是先做错误归因，将错误映射成结构化类别。

例如当前支持：

- `nonexistent_method`
- `invalid_overload_call`
- `invalid_constructor`
- `missing_import`
- `incompatible_types`
- `undeclared_variable`
- `invalid_generic_syntax`
- `nonexistent_type`
- `non_static_context`
- `void_assignment`

### 为什么要这样做

因为：

- 原始错误日志很长、噪音大
- 直接喂给模型，不容易稳定收敛

结构化后：

- 系统能把错误变成清楚的规则
- 下一轮 prompt 更容易得到有用约束

### 举例

如果错误是：

- `cannot find symbol: variable xxx`

系统会归因为：

- `undeclared_variable`

然后提炼成规则：

- 所有变量必须先声明再使用

如果错误是：

- `no suitable method found for ...`

系统会归因为：

- `invalid_overload_call`

然后提炼成规则：

- 只使用文档中存在的 overload，并保持参数数量和类型正确

### 如何验证这一步已经完成

查看：

- `runtime_rules_1.txt`

如果里面出现的是“规则句子”而不是原始报错，就说明这一步已经生效。

---

## 10. runtime prompt 更新层

### 使用的代码

- `Fuzz4All/target/target.py`

### 这一步做了什么

系统把：

- 当前总结出的 `runtime_rules`
- 当前已有的 `SAFE` 样例

一起写回 prompt，生成：

- `runtime_prompt_1.txt`

### 为什么需要更新 prompt

因为如果一直用初始 prompt，模型会反复犯同样的错误。  
把错误规则写回 prompt，可以让下一轮生成更稳定。

### 如何验证这一步已经完成

查看：

- `prompts/runtime_prompt_1.txt`
- `prompts/runtime_rules_1.txt`

如果都生成了，说明反馈已经成功回写 prompt。

---

## 11. refresh 层

### 使用的代码

- `Fuzz4All/target/target.py`

### 这一步做了什么

系统会判断：当前反馈是否足够支持生成一个更好的新 prompt。

如果可以，就尝试 refresh，并记录：

- `refresh_status_round_*.txt`
- `refresh_decision_round_*.txt`
- `accepted_refreshed_prompt_round_*.txt`

### 为什么要 refresh

这一步的作用是：

- 不只是“在原 prompt 上加几条规则”
- 而是尝试重新组织 prompt，让后续生成更稳

### 如何验证这一步已经完成

查看输出目录的 `prompts/` 是否有：

- `refresh_status_round_1.txt`
- `refresh_decision_round_1.txt`

如果还有：

- `accepted_refreshed_prompt_round_*.txt`

说明 refresh 不只是尝试了，而且被接受了。

# 四、系统如何从错误日志自动总结下一轮约束

这是最容易被问的地方，下面单独讲清楚。

## 第一步：先收集原始错误日志

来源是：

- `javac` 编译错误
- `java` 运行错误

这些错误会先进入：

- `sanitized_message`

并写入 `mutation_manifest.jsonl`

---

## 第二步：把原始错误归到类别

系统在 `JAVA.py` 中通过字符串模式和正则，判断错误属于什么类型。

例如：

- `cannot find symbol: method xxx`
  - → `nonexistent_method`

- `cannot find symbol: variable xxx`
  - → `undeclared_variable`

- `no suitable method found for ...`
  - → `invalid_overload_call`

- `cannot find symbol: class xxx`
  - → `nonexistent_type`

- `void cannot be converted`
  - → `void_assignment`

- `cannot be referenced from a static context`
  - → `non_static_context`

---

## 第三步：把错误类别翻译成规则

系统不会把错误类别直接给模型，而是进一步翻译成人类可理解、模型也能理解的规则。

例如：

### `undeclared_variable`

会变成：

- 所有变量必须在同一方法中先声明再使用

### `invalid_overload_call`

会变成：

- 只使用文档中存在的 overload，保持参数个数和类型一致

### `nonexistent_type`

会变成：

- 只使用文档或标准 JDK 中存在的公共类型，不要臆造类名

### `void_assignment`

会变成：

- 不要把 void 返回方法链式赋值给变量

---

## 第四步：按类别再加一层分类规则

这一步很关键。

系统不仅给出通用错误规则，还会根据 `primary_tag` 追加分类规则。

例如：

### `CALLBACK`

会补充：

- 只使用文档化的 listener / add / remove / fire overload
- 不要发明 indexed/helper listener API

### `JVM_MGMT`

会补充：

- 使用标准 `ManagementFactory` 入口
- 使用标准 `javax.management` / `java.lang.management` 类型
- 不要发明 bean container / proxy / wrapper 类名

### `REFLECT`

会补充：

- receiver、成员名、可访问性变化必须符合反射 API 语义

### `FILE`

会补充：

- 路径对象和状态变化必须显式且类型正确

---

## 第五步：把规则写成 `runtime_rules_1.txt`

最后这些规则会去重、压缩，并输出成：

- `runtime_rules_1.txt`

这就是“系统自动总结出的下一轮约束”。

---

## 第六步：把规则交给下一轮 prompt

系统再把：

- `runtime_rules_1.txt`
- 当前 SAFE 样例

融合成：

- `runtime_prompt_1.txt`

之后下一轮生成用的就不再是原始 prompt，而是更新后的 prompt。

所以整个机制不是“日志只记录了”，而是：

- 日志
  → 错误类别
  → 规则
  → runtime prompt
  → 下一轮生成

这就是闭环真正成立的地方。

---

# 五、怎么判断系统“成功”

这个系统的成功不能只看一个 SAFE 率，需要分层判断。

## 1. 机制成功

只要满足：

- 分类命中正确
- `mutation_profile` 正确
- manifest 中出现分类专属 operator
- 生成了 `runtime_rules` 和 `runtime_prompt`

就说明机制层成功。

---

## 2. 类别成功

如果一个类：

- 原来长期 `SAFE = 0`
- 经过分类级规则、imports、filter、feedback 优化后
- 开始出现 `SAFE`

就说明这个类已经从崩溃态进入可用态。

---

## 3. 效果成功

效果层看的是：

- `SAFE` 比例
- 不同类别之间是否稳定
- 是否还在大量出现低级幻觉错误

一般可以这样理解：

- `SAFE > 50%`
  - 很强，很稳定
- `20% ~ 50%`
  - 可用
- `1% ~ 20%`
  - 已起量，但还弱
- `0%`
  - 还没拉起来

# 七、别人可能会问的问题与回答

## Q1：为什么不直接让大模型读文档生成代码，还要多一层分类？

回答：

因为不同 API 的风险点不同。  
如果没有离线分类，后续 mutation 只能是 generic 的，不能体现安全类、文件类、反射类、并发类的差异。  
分类层是让系统具备“按类别选择 fuzz 策略”的关键。

---

## Q2：为什么不直接把原始报错日志丢回模型？

回答：

因为原始报错日志噪音很大，不利于稳定优化。  
系统先把错误归成结构化类别，再翻译成规则句子，这样更适合写回下一轮 prompt。

---

## Q3：为什么说你做的是“分类驱动变异”，不是简单生成？

回答：

因为：

- 运行前已有 `primary_tag / mutation_profile`
- 运行中 manifest 能看到不同类别 operator
- 说明生成后的变异行为确实被类别控制了

所以不是“生成完随便改一下”，而是“按类别变异”。

---

## Q4：SAFE、FAILURE、ERROR 怎么理解？

回答：

- `SAFE`：编译、运行都通过
- `FAILURE`：编译失败
- `ERROR`：编译通过但运行时报错

在 fuzz 场景里，`ERROR` 不一定是坏事，因为它表示样本已经进入了更深层的执行路径。

---

## Q5：现在最主要的剩余工作是什么？

回答：

现在不是再大改架构，而是：

1. 最终统一回归验证 10 类代表 API
2. 继续做分类级质量收敛
3. 整理最终结果表和材料
