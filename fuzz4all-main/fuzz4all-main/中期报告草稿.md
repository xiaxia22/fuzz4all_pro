# 本科毕业设计（论文）中期报告草稿

题目：面向 JVM 差分测试的分类感知大模型增强 Fuzzing 方法研究与实现

## 一、选题背景

随着 Java 生态的持续扩展，JDK 核心 API 与第三方库的规模不断增大，传统依赖人工设计种子或通用随机变异的 Fuzzing 方法，越来越难以覆盖复杂 API 的真实使用边界。尤其是在 JVM 相关程序测试中，很多缺陷并不出现在简单语法层面，而是隐藏在资源生命周期、参数边界、反射调用、并发交互、文件状态切换等深层语义路径中。

近年来，大语言模型在代码生成方面展现出较强能力，可以根据文档快速生成可运行程序，为自动化测试提供了新的思路。但单纯依赖大模型生成的测试代码，往往更倾向于“常规用法”和“安全写法”，难以主动探索高风险边界，因此容易出现“生成代码多、有效压力少”的问题。基于此，本课题在开题阶段提出将 API 文档学习、大模型提示词生成、五维分类感知和定制化变异策略结合起来，构建一套面向 JVM 场景的增强型 Fuzzing 框架，以提升测试样例的针对性、边界性和可解释性。

本课题的研究意义主要体现在两个方面：一是提升大模型生成测试样例的有效性，使其从“生成代码”进一步发展为“生成具有测试价值的代码”；二是为 JVM API 的自动化压力测试和差分测试提供一套具备分类认知、反馈优化和持续演化能力的工程化方法。

## 二、研究（设计）方案

### 2.1 总体方案框架

在开题方案的基础上，当前系统进一步细化为“离线分类建模 + 在线生成变异 + 反馈优化迭代”的闭环结构。总体流程如下：

```text
API Markdown 文档
        │
        ├── DeepSeek 文档学习与提示词蒸馏
        │           │
        │           ▼
        │      初始 Prompt
        │
        ├── 离线五维全息分类漏斗
        │      ├─ Package / Type / Interface
        │      ├─ Field / Method / Boundary
        │      └─ 输出分类标签与 mutation profile
        │
        ▼
Qwen/Ollama 代码生成模型
        │
        ▼
初始 Seed 程序生成
        │
        ▼
按 API 分类标签选择定制化变异策略
        │
        ▼
变异样例生成 + Repair/Filter 预处理
        │
        ▼
javac / JVM 编译运行验证
        │
        ├── SAFE / FAILURE / ERROR 日志记录
        ├── mutation_manifest.jsonl 样例证据链
        └── 结构化错误规则提取
                    │
                    ▼
            DeepSeek 运行期 Prompt 优化
                    │
                    └───────────────反馈到下一轮生成
```

### 2.2 方案简要说明

本课题方案主要由三个层次组成。

第一，构建 API 文档驱动的提示词生成模块。系统读取单个 API 的官方文档，由 DeepSeek 提取方法签名、边界条件、异常信息和典型使用方式，自动生成适合代码模型理解的 fuzzing prompt。这样可以避免手工为每个 API 编写提示词，提高规模化适配能力。

第二，构建五维全息分类漏斗与标签实现机制。针对 Java API，从包级语义、类型语义、接口约定、字段特征、方法特征及异常边界等多个维度进行综合打分，将 API 自动映射到 `RESOURCE`、`BUFFER`、`FILE`、`CONCURRENT`、`SECURITY`、`REFLECT`、`TIME`、`NETWORK`、`UTILITY` 等分类标签，并进一步映射为运行时可执行的 mutation profile。

第三，构建分类感知的混合变异与反馈优化闭环。Qwen/Ollama 根据 prompt 生成初始 seed 代码，系统再根据当前 API 的分类标签自动选择对应变异算子，对 seed 进行边界性改写。同时，编译运行日志会被结构化提取为规则，再由 DeepSeek 反向优化 prompt，形成“文档学习、生成、变异、验证、反馈、再生成”的闭环。

## 三、已经完成的任务

### 3.1 完成了 API 文档体系梳理与目标配置重构

前期首先对项目原有配置方式进行了梳理。原始 Fuzz4All 项目更偏向语言特性或少量手工目标的测试组织方式，不适合当前“以 API 文档为核心单位”的研究需求。针对这一问题，已将 `config/documentation` 中的内容统一替换为单个 API 的文档描述，并批量生成与之对应的 `config/targeted` 目标配置，使“一个文档对应一个 target”成为系统的基本组织方式。

在这一过程中，已经完成了全量 Java API 文档与 target 配置的对应化处理，配置文件命名与 API 名称保持一致，便于后续实验记录、批量运行与结果追踪。新的 target 配置中，不仅保留了原有的语言、输入提示和模型参数，还增加了分类结果路径、运行时反馈参数、变异预算、变异算子数量上限等字段，为后续分类驱动变异奠定了基础。

这一部分工作的完成，意味着项目已从“少量示例驱动”过渡到“面向大规模 API 文档驱动”的组织方式，满足了开题中提出的“基于 API 文档自动构建测试目标”的要求。

### 3.2 完成了五维全息分类漏斗的离线实现与分类结果落地

围绕开题提出的“五维全息分类漏斗”思路，当前已经完成离线分类脚本与分类结果输出。分类过程中综合考虑了以下几个维度：包路径、类型名称、接口语义、字段特征、方法特征以及异常/边界描述，并通过规则打分方式确定 API 的主标签和多标签集合。

目前已经完成对全部 Java API 文档的离线分类，实际得到 4410 个 API 的分类结果，并输出为 JSON、CSV 和汇总统计文件。这些结果不仅记录了每个 API 的 `primary_tag` 和 `all_tags`，还保留了分类得分与命中证据，便于后续分析和人工复核。

从目前结果看，像 `BufferedOutputStream`、`BufferedInputStream` 一类 API 能够稳定归入 `RESOURCE` 或 `BUFFER` 相关标签，`Thread` 能够映射到 `CONCURRENT`，`File` 能够映射到 `FILE`，`Cipher` 能够映射到 `SECURITY`，说明分类漏斗已经能够对主要 Java API 风险类型形成有效区分。相比只依赖文件名或简单关键字分类的方式，该方法更贴近开题中“多维融合判定”的设计初衷。

### 3.3 完成了分类标签到运行时 mutation profile 的映射机制

在离线分类完成后，课题进一步将分类结果接入运行时测试流程，实现了“标签驱动变异策略选择”的关键一环。当前系统已经建立了从 `primary_tag`、`all_tags` 到 `mutation_profile` 的映射关系，并补齐了多个组合画像，包括 `resource_buffer_batch`、`file_path_state`、`concurrent_sequence`、`security_provider_flow`、`reflection_dispatch`、`time_boundary_mix`、`network_endpoint_state` 等。

同时，原先只支持少数类别的 profile loader 已经被扩展为可以覆盖更多标签，新增并打通了 `UTILITY`、`JVM_MGMT`、`MARK_SUPPORT`、`GENERIC` 等类别。这意味着，当系统运行到某个 API 时，已经不再是统一地进行通用变异，而是能够依据离线分类结果自动激活当前 API 最匹配的一组风险画像和优先算子。

这一部分工作的价值在于，它真正把“分类结果”从离线分析信息变成了在线测试决策依据，使五维分类不再停留在描述层面，而开始对样例生成行为产生直接影响。

### 3.4 完成了多类定制化变异算子的扩展与接入

根据前期分类结果和开题中提出的“定制化边界打击”思路，当前已经在 Java 变异模块中实现并补充了多类具有针对性的变异算子。相较于原有以通用变异为主的方式，目前变异器已经能够围绕不同类别的 API 施加差异化干预。

例如，在 `FILE` 类别中，已经支持路径穿越、文件存在状态切换、读写权限切换等变异；在 `CONCURRENT` 类别中，已经支持线程让步、线程包装、同步信号序列等变异；在 `SECURITY` 类别中，已经支持算法边界、Provider 变异、密钥材料边界变异；在 `REFLECT` 类别中，已经支持可访问性切换、成员名称变异、接收者对象异常替换；在 `TIME`、`NETWORK`、`CALLBACK`、`UTILITY`、`JVM_MGMT`、`MARK_SUPPORT` 等类别中，也已补充相应的定制算子。

在实现这些变异算子的同时，还配套加入了修复与过滤机制，例如自动修复 byte 越界字面量、补齐必要 import、过滤明显幻觉式 API 使用、过滤结构不完整样例等。这样做的目的，是在保持边界攻击性的同时，尽量减少无效语法错误样例，提高变异后的可编译率与测试价值。

从系统结构上看，这部分工作已经基本完成了开题中“分类感知变异引擎”的初步实现，且已不再局限于单个示例 API。

### 3.5 完成了基于编译日志的反馈规则提取与 Prompt 演化闭环

除离线分类与变异策略外，当前系统已接入运行期反馈机制。具体而言，系统在编译阶段会对 `javac` 返回的错误日志进行归一化处理，将“方法签名不匹配”“类型不兼容”“缺少导入”“非法构造器调用”“反射 API 误用”等错误抽象成可复用规则，再动态写回到运行时 prompt 中。

为配合这一机制，Java 目标侧原先针对单个 API 的专用错误规则已被泛化为“目标 API + 主分类标签”共同驱动的规则生成方式。这样，当测试对象从 `BufferedOutputStream` 切换到 `Cipher`、`Thread`、`Method`、`File` 等 API 时，系统能够依据其自身分类特征，给出更贴合当前 API 的错误规避规则和安全样例参考。

此外，系统已经支持在运行中定期刷新 prompt，将积累的失败规则和成功样例交给 DeepSeek 进行二次 prompt 优化，从而逐步减少低层语法错误，提高后续样例的有效性。这说明项目已不仅能“生成和变异代码”，而且开始具备“根据反馈持续优化生成策略”的演化能力，基本实现了开题阶段设定的反馈闭环目标。

### 3.6 完成了阶段性样例验证与证据链记录机制

在实验侧，当前已经基于 `BufferedOutputStream` 等代表性 API 完成了阶段性运行验证，并能够在输出目录中生成 `log.txt`、`log_generation.txt`、`log_validation.txt`、`runtime_prompt_x.txt`、`runtime_rules_x.txt`、`mutation_manifest.jsonl` 等文件。特别是 `mutation_manifest.jsonl` 的引入，使每个样例都具备可追溯的“父种子—使用算子—分类标签—编译结果”证据链，便于后续分析哪一类 API 使用了哪一种策略、命中了哪些错误类型。

目前阶段性结果表明，系统至少已经在以下几个方面实现了预期能力：一是能够依据 API 文档自动生成 fuzzing prompt；二是能够按 API 的分类标签选择不同变异算子；三是能够记录编译反馈并更新运行时规则；四是能够将成功样例和失败规则用于后续 prompt 演化。这些结果说明课题已经从“方案设计”进入到“核心链路已接通、正在扩展类别和提升稳定性”的实施阶段。

## 四、存在问题及措施

### 4.1 问题一：运行成本较高，初始化与刷新阶段耗时明显

当前系统在 prompt 评估和运行期 refresh 阶段，需要多次调用大模型生成候选样例，并逐个进行编译验证。当使用本地 Ollama 模型或远程 API 时，这一过程开销较大，容易导致初始化较慢、整体运行周期较长。

措施：后续将进一步拆分“正式 fuzzing 批次”和“prompt 评估批次”，缩小验证 batch 大小，减少 refresh 候选数量，并仅在规则变化明显时触发 prompt refresh，以降低整体运行成本。

### 4.2 问题二：分类虽已覆盖全量 API，但部分标签仍需更多样例验证

当前离线分类已经完成全量覆盖，运行时标签和画像映射也已打通，但不同类别之间的实际变异效果仍存在差异。部分新补充标签虽然已具备对应算子，但还需要更多 API 样例来验证其稳定性和有效性。

措施：下一阶段将选取 `SECURITY`、`FILE`、`CONCURRENT`、`REFLECT`、`NETWORK` 等代表性类别开展专项验证，分析不同标签下的编译通过率、错误类型分布和有效边界触发情况，进一步修正画像和算子优先级。

### 4.3 问题三：差分测试场景尚未全面展开

当前工作重心主要集中在“分类驱动生成与变异闭环”的搭建和单目标验证，虽然已经具备面向 JVM 差分测试扩展的基础，但针对多版本 JDK 或多运行时环境的系统性差分实验还未全面完成。

措施：后续将在现有样例生成与日志记录机制基础上，接入多版本 JDK 编译运行环境，逐步开展跨版本差异检测实验，并整理出具有代表性的差分案例，为论文后续实验部分提供实证支撑。
