> 基于 `outputs/targeted7` 的 10 个代表 API 实际运行数据制定。  
> 目标不是再修单个 API，而是提升弱类与波动类在“文档生成 → 分类变异 → 编译/运行反馈 → prompt 更新”闭环中的**种子稳定性**。  
> 分析日期：2026-04-30

---

### 1.2 根因分类

当前弱类与波动类的问题，主要可归为四类：

#### 根因 A：Import 注入规则过弱

当前 [target.py](D:\same-fuzz\fuzz4all-main\fuzz4all-main\Fuzz4All\target\target.py:411) 的 `_ensure_java_import()` 仍然基于：

```python
signal = import_name.split(".")[-1] + "."
if signal not in code or f"import {import_name};" in code:
    return code
```

这只能在代码出现 `ClassName.` 形式时触发导入；但很多 Java 合法用法是：

```java
MemoryMXBean bean = ManagementFactory.getMemoryMXBean();
```

这里 `MemoryMXBean` 作为类型声明出现，不带点号，因此不会触发自动导入。

受影响明显的类：

- `ManagementFactory`
- `Method`
- `PropertyChangeSupport`
- 部分 `File`

---

#### 根因 B：LLM 幻觉 API / 幻觉变量

这一类是当前最普遍的失败源之一：

- 发明不存在的方法
- 使用错误的重载
- 生成并未声明的变量
- 把辅助类、注解、泛型写成非法 Java 形式

典型表现：

- `ManagementFactory`：发明不存在的 bean/query 方法
- `PropertyChangeSupport`：监听器重载、proxy、变量声明不稳定
- `Method`：注解、泛型、反射接收者与签名错位

---

#### 根因 C：明显非法 Java 语法没有被前置过滤

这类错误理论上可以在进入编译器之前就拦掉，例如：

- `\x00` / `\xFF` 这类非 Java 转义
- `default void ...` 写在普通类中
- 泛型参数位置错误
- 某些无条件死循环/极端等待路径

当前系统已有 `filter_rules` 和 `_is_viable_generated_code()`，但对弱类和波动类的非法模式拦截仍不够。

---

#### 根因 D：弱类种子质量仍然不足

即使分类和变异因子都已经命中正确画像：

- `CALLBACK -> callback_registration_flow`
- `JVM_MGMT -> jvm_mgmt_runtime_state`
- `REFLECT -> reflection_dispatch`

这些类仍可能因为 seed 本身不稳定而崩掉。  
这说明当前主要问题已经不是“分类失效”，而是：

- prompt 约束不够细
- 错误归因还不够精确
- repair/filter/import 体系还不够强

---

## 二、改进原则

本阶段的优化必须遵守两条原则：

1. **优先做分类级通用修复**
   - 改 `CALLBACK` 类，而不是只改 `PropertyChangeSupport`
   - 改 `JVM_MGMT` 类，而不是只改 `ManagementFactory`
   - 改 `RESOURCE` 类，而不是只改 `BufferedInputStream`

2. **必要时允许代表 API 做更强约束**
   - 分类级模板负责“通用可迁移”
   - 代表 API 的 YAML trigger 负责“当前验证样例更快起量”

---

### 第一层：代码层即时修复

这些修改对系统整体最有帮助，而且不依赖重新训练模型。

#### Fix-1：修复 `_ensure_java_import` 的触发条件

**文件：** `Fuzz4All/target/target.py`

当前问题：

- 只检查 `ClassName.`，漏掉类型声明场景

建议修改为：

```python
import re

class_name = import_name.split(".")[-1]
if f"import {import_name};" in code:
    return code
if not re.search(rf"\b{re.escape(class_name)}\b", code):
    return code
```

这条是**当前最值得优先落地**的修复之一。

#### Fix-3：为弱类和波动类补前置过滤规则

#### Fix-4：继续细化 Java 错误归因

**文件：** `Fuzz4All/target/JAVA/JAVA.py`

当前状态：

- 已经拆开了 `nonexistent_method` 和 `invalid_overload_call`
- 但还可以继续补：
  - `undeclared_variable`
  - `invalid_generic_syntax`

建议新增：

```python
if "cannot find symbol" in normalized and re.search(
    r"symbol:\s+variable\s+[A-Za-z_]", normalized
):
    categories.append("undeclared_variable")

if "'(' expected" in normalized and "generic" in normalized.lower():
    categories.append("invalid_generic_syntax")
```

并给出更明确的反馈规则：

```python
if "undeclared_variable" in categories:
    rules.append(
        "Declare all variables before using them; do not reference listener, proxy, or event objects that are not created in the same method."
    )
if "invalid_generic_syntax" in categories:
    rules.append(
        "Place type parameters before the return type: write `<T> void foo(T arg)`, not `void foo<T>(T arg)`."
    )
```

### 第二层：Prompt 与参数稳定性改进

这层不是“通用逻辑 bug 修复”，而是让弱类别在实验中更稳。

#### Fix-5：区分“调试加速版”和“稳定性验证版”参数

当前真实状态：

- 你现在为了缩短启动时间，弱类代表 target 用的是**调试加速版**

例如当前弱类代表 target 已经是：

- `autoprompt_candidates: 1`
- `autoprompt_validation_batch_size: 2`
- `autoprompt_validation_max_length: 192`
- `runtime_refresh_validation_batch_size: 2`
- `runtime_refresh_validation_max_length: 192`
- `prompt_refresh_interval: 2`

所以这里不能简单写成“统一改成 5”。

应该拆成两套：

**调试加速版**
- 适合快速迭代
- 启动快
- 噪音较大

**稳定性验证版**
- 适合弱类正式复测
- 可以考虑：
  - `prompt_refresh_interval: 5`
  - `autoprompt_validation_batch_size: 5`

建议顺序：

1. 继续用调试加速版找问题
2. 等 import/filter/feedback 逻辑补完后
3. 再把弱类切到稳定性验证版

---

#### Fix-6：补强弱类代表 API 的 trigger

这部分允许保留代表 API 的更强提示，但要明确它只是**代表 API 的验证增强**，不是通用类替代物。

**ManagementFactory 代表 trigger**

```text
Please create a very short, self-contained Java program that:
1. Imports java.lang.management.* at the top
2. Only calls documented static factory methods such as
   ManagementFactory.getMemoryMXBean(), getThreadMXBean(),
   getClassLoadingMXBean(), getRuntimeMXBean(), getCompilationMXBean()
3. Uses returned MXBeans to query one or two documented properties
4. Avoids invented management helper methods or non-JDK wrappers
```

**PropertyChangeSupport 代表 trigger**

```text
Please create a very short, self-contained Java program that:
1. Creates a PropertyChangeSupport(source) object where source is a simple Object
2. Declares listener objects locally
3. Uses documented add/remove/fire/hasListeners overloads only
4. Avoids indexed helper methods, proxy-heavy constructions, and undeclared variables
```

## 七、总结

当前系统的主要矛盾已经不是“分类失效”，而是：

1. 弱类的 import / 过滤 / 错误归因仍不够强
2. seed 质量在弱类和波动类上仍然不稳定
3. prompt 与 refresh 已经形成闭环，但弱类还没从闭环中真正获益

因此，下一阶段的工作重点应当是：

- 先补通用 import / filter / feedback 机制
- 再提升弱类代表 API 的种子稳定性
- 最后再用 10 类代表集做统一回归验证

这条路线和当前代码状态、当前 `targeted7` 数据，以及整体设计目标是对齐的。
