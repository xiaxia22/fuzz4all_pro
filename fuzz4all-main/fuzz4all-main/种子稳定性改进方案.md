### 1.2 根因分类

通过逐条分析 manifest 确定以下四类根因：

#### 根因 A：Import 注入失效（主因之一）

当前 `_ensure_java_import` 的触发条件是 `ClassName.` 出现在代码中（带点号），而**类作为变量类型声明时不带点号**，例如：

```java
MemoryMXBean memoryMXBean = ManagementFactory.getMemoryMXBean();
//  ↑ 这里没有点号，_ensure_java_import 判断不需要导入 → 编译失败
```

**受影响 API：**
- `ManagementFactory`：60 条 FAILURE 中有 60 次 `MemoryMXBean`/`ClassLoadingMXBean`/`RuntimeMXBean` 类符号未找到
- `Method`：`Map`、`List` 未导入（`java.util`）
- `PropertyChangeSupport`：`PropertyChangeEvent` 偶发未导入

#### 根因 B：LLM 幻觉 API（最普遍的失败源）

LLM 生成了**不存在的方法、构造函数或类**：

| API | 幻觉内容 | 触发量 |
|-----|---------|--------|
| ManagementFactory | `getTotalCompliationTime()` (typo)、`totalCommittedBytes()` | ~15条 |
| PropertyChangeSupport | `addIndexedPropertyChangeListener()`（不存在）、`PropertyChangeListenerProxy(String, String, Object, null)`（构造函数参数全错） | ~10条 |
| Method | `@MyAnnotation`（自定义注解）、`default void`（写在 class 而非 interface 中）、`public void genericMethod<T>()` 泛型位置错误 | ~24条 |
| PropertyChangeSupport | 使用了 `listener1`, `listener2` 等**从未声明**的变量 | ~21条 |

#### 根因 C：非法 Java 语法（可过滤）

LLM 生成了 Java 不支持的语法，但这类错误**可以在运行前用正则过滤掉**：

| API | 非法语法 | 样本量 |
|-----|---------|--------|
| File | `"\x00\xFF"` —— Python/C 风格十六进制转义，Java 不支持 | 17/50 条 |
| Method | `default void foo()` —— 只能用于接口，写在普通类中报错 | 12/60 条 |
| Method | `public void foo<T>()` —— 泛型参数位置写错（应为 `<T> void foo()`） | 6/60 条 |
| Thread | `while(true)` 或 `for(;;)` —— 无条件死循环，导致超时 | ~3/50 条 |

#### 根因 D：Prompt 过早不稳定

- 所有 10 个 API 中，**只有 BufferedInputStream 有 1 次 prompt 接受刷新**，其余 9 个均为 0 次
- `prompt_refresh_interval: 2` 意味着每 2 个 batch 就刷新一次，但刷新需要 SAFE 样本或规则信号才能评分，早期全为 FAILURE 时评分为 0 → 刷新总是被拒绝 → 反馈闭环空转
- `autoprompt_validation_batch_size: 2`：仅用 2 个样本评分初始 prompt，噪音极大

## 二、改进方案

按实现成本由低到高分为三层。

---

### 第一层：代码层面的即时修复（最高优先级）

这些改动不需要重新训练模型或调整 API，修改代码即可生效。

#### Fix-1：修复 `_ensure_java_import` 的触发条件

**文件：** `Fuzz4All/target/target.py`

**问题：** 当前使用 `ClassName.` 信号（类名后跟点号）判断是否需要导入，但变量类型声明场景无法触发。

**修改：** 改用单词边界正则 `\bClassName\b` 检查类名是否出现在代码中：

```python
# 改前：只检查 "ClassName."
signal = import_name.split(".")[-1] + "."
if signal not in code or f"import {import_name};" in code:
    return code

# 改后：检查类名作为完整单词出现（覆盖变量类型声明场景）
import re
class_name = import_name.split(".")[-1]
if f"import {import_name};" in code:
    return code  # 已有导入，跳过
if not re.search(rf'\b{re.escape(class_name)}\b', code):
    return code  # 类名没有出现，不需要导入
# 插入 import
```

**预期效果：** ManagementFactory SAFE 率从 0% → 约 35%+（大部分失败仅仅是缺少 MXBean 类导入）

#### Fix-3：为各 profile 增加 filter_rules 拒绝非法模式

**文件：** `Fuzz4All/mutation/profile_loader.py`

在 `_profile_rejects_generated_code` 过滤阶段提前拦截已知错误模式，**避免将无效 seed 送入编译器**：

**FILE profile 新增过滤：**
```python
"reject_patterns": [
    r'\\x[0-9A-Fa-f]{2}',       # \x00 风格 —— Python/C 风格，Java 不支持
    r'new\s+File\s*\(\s*null\s*,', # new File(null, "...") —— 两个构造函数歧义
],
```

**REFLECT profile 新增过滤：**
```python
"reject_patterns": [
    # 自定义注解（标准注解以外的 @大写开头 词）
    r'@(?!Override|SuppressWarnings|Deprecated|FunctionalInterface|SafeVarargs|Serial)\b[A-Z][a-zA-Z]+\b',
    r'\bdefault\s+(?:void|int|long|boolean|String|double|float)\s+\w+\s*\(',  # default方法写在class里
    r'public\s+\w+\s+\w+\s*<[A-Z]>\s*\(',  # 泛型参数位置错误 foo<T>() 应为 <T> foo()
],
```

**CONCURRENT profile 新增过滤：**
```python
"reject_patterns": [
    r'while\s*\(\s*true\s*\)',       # 无条件死循环 → 超时
    r'for\s*\(\s*;;\s*\)',           # for(;;) 死循环 → 超时
    r'Thread\.sleep\s*\(\s*Long\.MAX_VALUE',  # sleep(MAX) → 超时
],
```

**CALLBACK profile 新增过滤：**
```python
"reject_patterns": [
    r'addIndexedPropertyChangeListener',    # 不存在的方法
    r'removeIndexedPropertyChangeListener', # 不存在的方法
    # 自定义注解
    r'@(?!Override|SuppressWarnings|Deprecated)\b[A-Z][a-zA-Z]+\b',
],
```

#### Fix-4：为 JAVA.py 的错误分类新增两个常见模式

**文件：** `Fuzz4All/target/JAVA/JAVA.py`

在 `extract_javac_error_categories` 中补充两个高频错误类型，让反馈规则更精准：

**补充错误类型：**

```python
# 未声明变量（LLM 在代码中引用了从未声明的变量）
if "cannot find symbol" in normalized and re.search(
    r"symbol:\s+variable\s+[A-Za-z_]", normalized
):
    categories.append("undeclared_variable")

# 泛型类型参数位置错误（如 void foo<T>() 应为 <T> void foo()）
if "'(' expected" in normalized and re.search(r"generic", normalized.lower()):
    categories.append("invalid_generic_syntax")
```

**在 `map_error_categories_to_rules` 中对应的规则：**

```python
if "undeclared_variable" in categories:
    rules.append(
        "Declare all variables before using them; do not reference identifiers "
        f"(listener, proxy, event objects) that are not created in the same method."
    )
if "invalid_generic_syntax" in categories:
    rules.append(
        "Place type parameters before the return type: write `<T> void foo(T arg)`, "
        "not `void foo<T>(T arg)`."
    )
```

### 第二层：Prompt 质量改进（中等优先级）

#### Fix-5：调整 0% SAFE API 的 Prompt 稳定性参数

**文件：** 各 YAML 配置

| 参数 | 当前值 | 建议值 | 原因 |
|------|--------|--------|------|
| `prompt_refresh_interval` | 2 | **5** | 每 2 轮就刷新，早期 FAILURE 占主导时信号嘈杂，刷新永远被拒绝，浪费 API 调用 |
| `autoprompt_validation_batch_size` | 2 | **5** | 只用 2 个样本打分，一半概率全失败 → 评分 0 → 选了最差 prompt |

这两项修改对 ManagementFactory 和 PropertyChangeSupport 尤为关键。

#### Fix-6：精化 `trigger_to_generate_input`（针对 0% SAFE API）

当前 YAML 的 trigger 注释对 ManagementFactory 描述的是理想目标，但没有给出约束。

**ManagementFactory 建议的 trigger：**
```
/* Please create a very short, self-contained Java program that:
   1. Imports java.lang.management.* at the top
   2. Only calls documented static factory methods: ManagementFactory.getMemoryMXBean(),
      getThreadMXBean(), getClassLoadingMXBean(), getRuntimeMXBean(), getCompilationMXBean()
   3. Uses the returned MXBean objects to query one or two documented properties
   4. Wraps everything in try-catch
   Do NOT invent method names. Do NOT use PlatformMBeanProxy directly. */
```

**PropertyChangeSupport 建议的 trigger：**
```
/* Please create a very short, self-contained Java program that:
   1. Creates a PropertyChangeSupport(source) object where source is a new Object()
   2. Declares anonymous PropertyChangeListener objects using lambda: e -> {}
   3. Uses ONLY: addPropertyChangeListener, removePropertyChangeListener,
      firePropertyChange(String, Object, Object), hasListeners(String)
   4. Does NOT use PropertyChangeListenerProxy, addIndexedPropertyChangeListener,
      or any methods not in the list above
   5. Declares all variables used */
```

### 第三层：架构级改进（长期，可选）

#### Fix-7：文档驱动的方法白名单

思路：从 `config/documentation/` 中的 API 文档提取公开方法签名列表，在 prompt 更新时作为"合法方法白名单"注入约束。

```python
# profile_loader.py 新增
def extract_public_methods_from_doc(doc_path: str) -> List[str]:
    """从 Markdown 文档中提取 '### methodName' 或 '**methodName**(' 格式的方法名"""
    ...

# target.py 的 _create_feedback_prompt 中
if self.allowed_methods:
    rules.append(
        f"Only call these documented methods of {target_api}: "
        + ", ".join(f"`{m}()`" for m in self.allowed_methods[:10])
    )
```

#### Fix-8：种子模板（Golden Seed）机制

对于长期 0% SAFE 的 API，提供人工编写的"黄金种子"（已知能编译的最简程序），作为：
1. autoprompt 评分时的参照基准
2. 变异的高质量起始点

```yaml
# YAML 中新增字段
target:
  path_golden_seeds: config/golden_seeds/java.management_ManagementFactory.java
```

实现：在 `initialize()` 中若 golden seed 存在，优先用它评分 prompt 候选，再用它作为第一批变异 seed。

#### Fix-9：Prompt Refresh 窗口加权

当前 `batch_feedback_window` 存储最近所有规则，刷新时等权处理。
改进：对高频（重复出现 3次+）失败规则赋予更高权重，确保"undeclared_variable"这类系统性错误在 prompt 中有明确约束。

## 五、验证方法

每次修改后，用以下命令跑 5 样本快速验证（约 5~10 分钟）：

```bash
# 验证 ManagementFactory（最典型的修复案例）
python Fuzz4All/fuzz.py \
  --config config/targeted/java.management_java_lang_management_ManagementFactory.yaml \
  main_with_config \
  --folder outputs/verify_mgmt \
  --model_name Qwen/Qwen2.5-Coder-7B-Instruct

# 查看 SAFE 率
python -c "
import json
total=safe=0
with open('outputs/verify_mgmt/mutation_manifest.jsonl') as f:
    for line in f:
        d=json.loads(line); total+=1
        if d['compile_result']=='SAFE': safe+=1
print(f'SAFE {safe}/{total} = {safe/total:.0%}')
"
```

期望：ManagementFactory SAFE 率从 0% 提升至 30%+，File 从 6% 提升至 25%+。
